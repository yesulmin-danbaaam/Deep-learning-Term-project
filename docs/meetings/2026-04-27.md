## 2. 주요 논의 내용

### 2.1 프로젝트 방향성

프로젝트는 단순 아이디어 제안이 아니라, 실제로 학습 또는 실험을 수행할 수 있어야 한다는 의견이 나왔다.

교수님이 흥미로워할 만한 신박한 주제를 선정하되, 수업에서 배운 CNN, RNN, Transformer 등의 개념을 적용하고 실험 결과를 보여주는 방향이 적절하다는 의견이 있었다.

특히 실제 학습을 위해서는 데이터셋이 필요하므로, **주제 선정 시 데이터 확보 가능성이 가장 중요하다**는 점이 강조되었다.

---

### 2.2 Fine-tuning vs Scratch 학습

Scratch부터 Transformer를 학습하는 방식과, 이미 학습된 SLLM을 LoRA/QLoRA 방식으로 fine-tuning하는 방식이 비교되었다.

Scratch 학습은 교육적으로 의미가 있지만 데이터와 학습 비용이 많이 필요하므로, 실제 성능을 내기는 어렵다는 의견이 있었다.

반면 fine-tuning은 상대적으로 적은 데이터로도 특정 태스크에 맞춘 결과를 만들 수 있어 현실적인 대안으로 언급되었다.

| 방식               | 장점                     | 단점                    |
| ---------------- | ---------------------- | --------------------- |
| Scratch 학습       | 구조 이해에 좋음, 수업 내용 반영 강함 | 데이터/시간/GPU 부담 큼       |
| LoRA fine-tuning | 현실적, 빠르게 결과 가능         | 모델 구조 기여도가 약해 보일 수 있음 |
| Scratch + 작은 모델  | 교육용으로 적합               | 품질은 낮을 가능성 높음         |

---

### 2.3 교수님 평가 성향 관련 논의

이전 수업 경험상 교수님은 단순히 결과만 보는 것이 아니라, 주제 선정과 실험 과정의 논리성을 중요하게 본다는 의견이 있었다.

예상 질문은 다음과 같다.

* 왜 이 모델을 사용했는가?
* 왜 이 평가 지표를 사용했는가?
* 왜 이 데이터셋을 선택했는가?
* 해당 구조가 문제 해결에 논리적으로 맞는가?

따라서 주제를 정할 때는 “재미”뿐만 아니라, **데이터 → 모델 → 실험 → 결과 해석**의 흐름이 논리적이어야 한다는 점이 정리되었다.

---

### 2.4 일정 관련 논의

프로젝트 발표 및 제출까지 약 한 달 정도 남아 있어, 너무 무거운 구현은 위험하다는 의견이 있었다.

영상 촬영, 데모 제작, 학습, 데이터셋 정리까지 고려하면 아이디어 선정과 데이터 확보를 빠르게 마쳐야 한다는 점이 강조되었다.

# 5. 회의에서 나온 프로젝트 아이디어

## 5.1 SLLM Tool Calling / MCP Agent

자연어 요청을 MCP Tool Call로 매핑하는 모델을 만드는 아이디어가 논의되었다.

예를 들어 사용자가 “브라우저 열어서 네이버 검색해줘”라고 입력하면, 모델이 Playwright MCP의 적절한 툴을 선택하고 필요한 인자를 생성하는 방식이다.

| 항목  | 내용                                                 |
| --- | -------------------------------------------------- |
| 목표  | 자연어 요청 → 적절한 MCP Tool 선택                           |
| 모델  | 작은 Transformer 또는 SLLM fine-tuning                 |
| 데이터 | Claude Code 사용 로그, 공개 tool-calling 데이터셋, 자체 제작 데이터 |
| 우려점 | 데이터 확보, 학습 난이도, 4주 내 구현 가능성                        |

---

## 5.2 작은 Transformer / Guppy LM

아주 작은 Transformer 모델인 Guppy LM과 같은 학습용 모델 구조를 가져와, 여기에 레이어를 추가하거나 구조를 수정해 실험하는 방향이 제안되었다.

이 방식은 성능 자체보다는 “우리가 모델 구조를 이해하고 수정했다”는 점을 보여줄 수 있다는 장점이 있다.

다만 성능이 제한적일 수 있으므로, 필요하면 작은 pretrained 모델을 가져와 fine-tuning하는 방식도 함께 고려하기로 했다.

---

## 5.3 비전 / 모션 캡처

모션 캡처나 카메라 기반 비전 프로젝트도 논의되었다.

다만 비전 쪽은 데이터셋이 생각보다 제한적이고, 직접 수집해야 할 가능성이 있어 부담이 있다는 의견이 있었다.

---

## 5.4 온디바이스 / 상품 객체 인식

온디바이스 AI, 상품 인식, 바코드, 가방/넥타이 분류 등 기존 모델을 fine-tuning하는 방향도 언급되었다.

기존 모델을 활용하면 구현 안정성은 있지만, 프로젝트 주제의 참신함을 어떻게 확보할지가 추가 과제로 남았다.

---

## 5.5 CommCoach / 문장 순화 AI

회사나 일상 대화에서 공격적인 표현을 더 부드럽게 바꾸는 아이디어도 다시 언급되었다.

예를 들어 “이거 왜 안 했어?” 같은 문장을 “진행 상황 공유 부탁드립니다”처럼 바꾸는 방식이다.

이 아이디어는 실생활성이 있고 데이터셋도 어느 정도 확보 가능하지만, 팀원 중 일부는 더 재미있는 주제를 원한다는 의견을 냈다.

---

## 6. 회의 결론

이번 회의에서는 프로젝트 주제를 최종 확정하지 않고, **데이터셋 확보 가능성과 구현 난이도를 기준으로 후보를 더 좁히기로 했다.**

- 다다음주(5/11)까지 각자 카카오톡에 아이디어를 내는 기간으로 하고 
- 아이디어 제안시 , **아이디어 (주제나 컨셉) + 대략적인 계획 + 실제 활용 데이터셋** 형식으로 제안하기로 했다.

특히 단순히 재미있는 아이디어보다, 실제 학습과 실험이 가능하고 교수님 질문에 논리적으로 답할 수 있는 주제를 선택하는 것이 중요하다는 데 의견이 모였다.
